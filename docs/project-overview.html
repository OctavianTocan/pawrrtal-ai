Architecture · Detail

What happens on each chat turn

End-to-end flow from user input to persisted response — same path for web and Telegram.

Web / Electron path

1
Frontend POST /api/chat
Sends { conversation_id, question, model_id? } with JWT cookie. TanStack Query opens an SSE connection and streams the response into the chat reducer.
2
Backend resolves conversation + model
Loads conversation row. Resolves model in priority order: request.model_id → conversation.model_id → default. Persists the model change if it differs.
3
History window fetched
Reads the last 20 messages (_HISTORY_WINDOW = 20) from chat_messages ordered by ordinal. Passed to the provider as history: [{role, content}]. The current question is not included in this slice — it's passed separately as question.
4
User row + assistant placeholder committed
Two rows inserted into chat_messages before streaming starts. The assistant row starts as assistant_status="streaming" with empty content. Conversation.updated_at is bumped via _touch_conversation() on both inserts so the sidebar re-sorts.
5
Workspace system prompt loaded
SOUL.md (who the agent is) and AGENTS.md (operating rules) are concatenated from the user's workspace root and passed as system_prompt to the provider. Missing files are silently skipped. Capped at 64 KB per file.
6
Provider streams events
The resolved provider (Gemini or Claude) calls the LLM and yields StreamEvent dicts (delta, thinking, tool_use, tool_result, done, error). Each event is fed through ChatTurnAggregator which builds up the final persisted shape.
7
Channel delivers events to the client
The SSEChannel encodes each event as an SSE frame. render_artifact tool calls are lifted out as a sibling artifact SSE event. send_message tool calls drain from a per-request queue after each provider event.
8
Assistant row finalized
In a finally block (always runs, even on disconnect): the assistant chat_messages row is updated with final content, thinking, tool_calls, timeline, thinking_duration_seconds, and assistant_status ("complete" or "failed").

Telegram path

1
aiogram dispatcher receives update
Bot receives the Telegram update via polling or webhook. Non-text messages (photos, stickers, etc.) are silently dropped at the _on_message handler gate.
2
Binding resolved
handle_plain_message looks up the Telegram user in channel_bindings → returns the Pawrrtal user_id and the stable Telegram conversation (created on first message). Unbound users get a nudge string; plain-text 8-char codes are auto-redeemed here too.
3
⏳ placeholder sent, previous stream cancelled
Bot answers with "⏳" immediately. Any existing asyncio.Task for this chat_id is cancelled (_running_tasks dict, process-local).
4
History fetched + messages persisted
Last 20 messages loaded from chat_messages. User message row + assistant placeholder inserted. Conversation.updated_at bumped. Committed before streaming starts. This makes the turn visible in the web UI immediately.
5
Provider streams via TelegramChannel
TelegramChannel.deliver() accumulates token deltas and calls bot.edit_message_text when ≥40 new chars have arrived or 3 s have elapsed. Maximum message length 4096 chars (Telegram limit); truncates rather than splits. "message is not modified" errors are swallowed.
6
Assistant row finalized
Same finalize_assistant_message call as the web path — always runs in a finally block. Full content, tool calls, and final status are persisted.
7
Auto-title (first turn only)
Gated by title_set_by IS NULL. Derives title from first user message (48-char max). Sets title_set_by = "auto". If inside a Telegram topic thread, also calls editForumTopic to rename the thread in the user's topic list.
Architecture · Detail

What the agent knows each turn

Context budget breakdown — what's in the system prompt, history, and tool surface.

Context Component Source Size / Behaviour
System prompt SOUL.md + AGENTS.md from workspace root Concatenated with --- separator. Each file capped at 64 KB. Missing files are silently skipped. Passed as system_prompt to provider.
Conversation history chat_messages table Last 20 messages (_HISTORY_WINDOW = 20) ordered by ordinal. Only role: "user" and role: "assistant" rows. Passed as history list. Current question passed separately.
Current question Request body / Telegram message text Passed directly as question arg to provider.stream(). Not included in the history slice for this turn.
Agent tools app.core.agent_tools.build_agent_tools() Workspace-scoped. Includes read_file, write_file, list_dir, render_artifact, send_message. Tools absent when no workspace exists (user hasn't completed onboarding).
Model identity Conversation.model_id → request override → default Resolved per turn. Default: gemini-3-flash-preview (web) / google/gemini-3-flash-preview (Telegram). Telegram supports /model <provider/id> to switch.
Context compaction: not yet implemented. The current history window is a hard truncation to the last 20 messages. Long conversations simply lose older messages. Options under consideration: (1) rolling summary — LLM-generated summary injected as a synthetic message before the window; (2) token-budget window — include as many messages as fit under a token ceiling; (3) semantic retrieval — embed all messages, retrieve top-k by cosine similarity; (4) provider-native memory — use previous_response_id (Responses API) or Claude's extended context. Decision pending; rolling summary is the recommended first step.
Architecture · Detail

Cross-surface conversation sync

How a Telegram conversation becomes visible on the web — and vice versa.

1
One conversation, multiple surfaces
Every Telegram user has a stable Conversation row with origin_channel = "telegram". This is the same table the web UI reads from. No separate Telegram-only store.
2
Messages persisted to shared table
Every Telegram turn writes user + assistant rows to chat_messages — the same rows the web /api/messages endpoint reads. Open the conversation on web and you see the full Telegram history.
3
Conversation list re-sorts on activity
Conversation.updated_at is bumped on every user message insert (_touch_conversation in crud/chat_message.py). The web sidebar orders conversations by updated_at DESC, so a Telegram message surfaces the conversation at the top of the list.
4
Telegram topic threads → stable sub-conversations
When Telegram Topics are enabled in a group, each message_thread_id gets its own Conversation row (keyed on telegram_thread_id). /new in a thread creates a fresh conversation scoped to that thread, leaving other threads untouched.
Real-time web update: The web frontend polls conversations via TanStack Query with a configured refetch interval. A message arriving on Telegram will appear in the web UI on the next poll cycle. True push (WebSocket or SSE for the conversation list) is a future enhancement.
Architecture · Detail

LLM provider layer

How models are resolved, configured, and streamed — and where Codex OAuth fits.

Provider Auth Endpoint / transport Notes
Gemini User API key (Fernet-encrypted in DB) google-generativeai SDK Default provider. Supports native tool use, thinking tokens, streaming.
Claude User API key (Fernet-encrypted in DB) Anthropic SDK via claude_provider.py Full tool use and extended thinking support.
OpenAI (image) Codex OAuth (ChatGPT account) chatgpt.com/backend-api/codex/responses Already wired for generate_image tool. Uses Codex CLI OAuth tokens, not API key credits.
OpenAI (text) — planned Codex OAuth (same flow) chatgpt.com/backend-api/codex/responses Design doc at docs/design/codex-oauth-text-provider.md. Confirmed endpoint via OpenAI developer docs + openai-oauth open-source proxy. Not yet implemented.
Codex OAuth endpoint: The correct endpoint for ChatGPT OAuth tokens is chatgpt.com/backend-api/codex/responses — not api.openai.com/v1/responses. The latter rejects ChatGPT OAuth tokens with a 401 (api.responses.write scope missing). The Codex-specific endpoint accepts the same tokens the Codex CLI uses and does not require API credits.
Settings · Integrations

Integrations section

The UI shell is live — no integrations are implemented in the backend yet.

UI shell
The Settings → Integrations page renders an integration list and an "Add Integration" modal. Both lists (YOUR_INTEGRATIONS, INTEGRATION_CATALOG) are empty arrays.
No fake data
Previously the catalog had mock "connected" Apple Calendar / Gmail / Drive entries to prototype the UI. These are removed — the page now shows a clear "Coming soon" notice with an empty state.
Add real integration
To wire up a real integration: implement backend OAuth flow, add a row to YOUR_INTEGRATIONS in catalog.ts, and remove the coming-soon notice. The UI components are all in place.
Custom MCP servers
The "Add Custom MCP" modal is already shipped and wired to the MCP server infrastructure. Users can add arbitrary MCP endpoints outside the curated catalog.
Features

What it does

Core capabilities shipped or in active development.

Real-time streaming
Chunk-by-chunk AI responses via Server-Sent Events with abort support per request.
Conversation management
Create, list, rename, and resume conversations with LLM-generated auto-titles.
Multi-model selection
Switch between Google Gemini and Claude models live. Telegram users use /model command.
Secure authentication
JWT tokens in httpOnly cookies. No tokens in localStorage. CSRF-safe by design.
User preferences
Custom system instructions, accent color, font size — stored in workspace preferences.toml.
API key vault
Bring-your-own keys for LLM providers, stored encrypted at rest with Fernet.
Telegram channel
Full two-way chat via Telegram. Conversations sync to web. Topic threads get their own conversation rows. /model, /stop, /new commands supported.
Electron desktop shell
Same Next.js frontend wrapped for desktop. IPC-bridged capabilities with web fallbacks.
Whimsy generator
Collision-safe procedural doodle placer for expressive, branded empty states.
Projects
Group conversations and knowledge under projects for organised workspaces.
Agent tools
Per-workspace file tools (read_file, write_file, list_dir), artifact rendering, and send_message for mid-turn push events.
Image generation
GPT image model via Codex OAuth. Tool call: generate_image. Delivers the image inline to the chat surface (web queue-drain or Telegram media send).

Quality & CI

How quality is enforced

Gates run on every PR to main. Self-hosted runner pool on VPS.

Biome
Lint + format for TypeScript/TSX. Zero lint suppressions without justification.
ruff + mypy
Python lint, format, and strict type checking via just check-all.
Sentrux
Architectural layer enforcement. Violations block PR merge. Baseline 6753/10000.
Stagehand E2E
LLM-driven browser tests via Stagehand v3 + GPT-4.1-mini. just stagehand-e2e.
File-line budget
500-line hard ceiling per .ts/.tsx/.py file. CI fails on overflow.
Design lint
bun run design:lint validates DESIGN.md token spec in CI on every PR.
