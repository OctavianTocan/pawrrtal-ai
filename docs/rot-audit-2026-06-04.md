## 0. Known architectural issues (diagnosed manually; excluded from the automated hunt to avoid duplication)

These three are the largest structural problems. They were identified by hand-reading the
code and were deliberately **excluded from the subagent rot-hunt** (so the appendix below does
not repeat them). They share one root pattern: *one interface pretending two archetypes are the
same*.

### 0.1 — The agent loop is actually TWO loops, and `loop.py` hides it
- `backend/app/agents/loop.py` `agent_loop()` is a pi-inspired, provider-agnostic loop that owns
  the turn lifecycle, sequential tool execution, the safety layer, and the permission gate.
- **Only raw-model providers run through it** — Gemini / xAI / litellm / opencode-go supply a
  `StreamFn` yielding `LLMEvent` (evidence: `litellm_provider.py:59` imports `agent_loop`;
  `gemini/messages.py` emits `LLMDoneEvent` into it).
- **Claude and Codex bypass it entirely.** `providers/claude/provider.py` runs the
  `claude-agent-sdk`'s own agentic loop via `resume=` with tools mounted as MCP servers
  (`ClaudeAgentOptions.mcp_servers`); `openai_codex` does the same via its SDK `thread_id`. So
  `loop.py`'s safety caps / permission gate / iteration limit **do not apply** to Claude/Codex
  turns — and nothing in `loop.py` says so.
- **Three event vocabularies:** `LLMEvent` (provider→loop), `AgentEvent` (loop→caller),
  `StreamEvent` (`providers/base.py`, provider→turn_runner). `AgentEvent` and `StreamEvent` are
  ~90% identical with a hand-written translation inside `GeminiLLM.stream`.
- **Confusing names:** `tool_call_end` fires when the *model* finished emitting the call's
  arguments, not when the tool finished executing (that's `tool_result`); `StreamFn` vs
  `AILLM.stream` (two provider streaming seams both called "stream"); `convert_to_llm` vs
  `transform_context` (both rewrite messages); `provider_state` (opaque, threaded through 4
  types under one blank name); `_StreamOutcome.events` is appended to but **never read** (dead).
- **Fix direction:** split into `ModelProvider` (raw, loop-driven, yields deltas) vs
  `AgentProvider` (owns its loop, yields turn events), both normalized to ONE outward event type
  (`AgentEvent` + `StreamEvent` merge), with declared capabilities instead of silent param-ignoring.

### 0.2 — `AILLM` is a "fat optional protocol" that leaks provider specifics upward
- `backend/app/providers/base.py` `AILLM.stream()` has one uniform signature, but every provider
  honors a **different subset** of its params — the docstrings literally say "providers that
  manage their own X may ignore this." A contract where half the params are ignore-by-convention
  is a suggestion, not a contract.
- The leak has escaped into the supposedly-shared pipeline:
  - `channels/turn_runner.py` `_guarded_stream` smuggles `codex_thread_id` / `agy_conversation_id`
    / `per_turn_context` through `**extra_kwargs`, conditionally, per provider; sets
    `provider_history = []` for codex/agy; special-cases `internal` events in
    `_handle_internal_provider_event` (codex_thread_created, agy_conversation_created).
  - `ChatTurnInput` carries provider-specific fields (`codex_thread_id`,
    `codex_thread_prompt_hash`, `codex_lightweight_prompt`, `agy_conversation_id`).
  - `factory.resolve_llm` is a per-provider construction if-ladder because no two providers
    construct alike.
- **Fix direction:** one `TurnRequest` object instead of 9 params + `**extra_kwargs`; move
  continuity to a provider-owned `ConversationStore` port; declare capabilities.

### 0.3 — The `Channel` contract is partly fictional and SSE-shaped
- `backend/app/channels/base.py`:
  - `Channel.receive()` is documented as responsibility #1 ("normalize the raw inbound event")
    but is **never declared on the Protocol and never implemented** by any channel; each channel
    normalizes inbound ad-hoc (`telegram/bot.py`, `google_chat/ingress.py`).
  - `ChannelResponse` is defined and exported from `__init__.py` and described as "the normalized
    outbound shape consumed by any channel" — but **nothing consumes it**; `deliver()` takes raw
    `StreamEvent`. It was meant to be the provider↔channel decoupling layer; because it's unused,
    every channel imports the 40-field `StreamEvent` kitchen-sink directly.
  - Stale docs: the module docstring lists Telegram as "Planned" and never mentions Google Chat;
    `registry.py` says `resolve_channel("telegram") # TelegramChannel (future)` — it's the
    largest channel.
- `deliver(stream) -> AsyncIterator[bytes]` is SSE-shaped. SSE yields real frames; **Telegram and
  Google Chat yield nothing** and end with a dead `return` / `yield  # pragma: no cover` purely to
  satisfy the async-generator shape.
- **Two opposite control flows behind one method:** web *pulls* bytes through a
  `StreamingResponse`; Telegram/GChat call `run_turn` themselves, discard the bytes
  (`async for _ in run_turn(...)`), and deliver as side-effects.
- **~100x size asymmetry:** web/SSE ≈ 101 lines; Telegram ≈ 10,000 lines across ~35 files
  (`bot.py` alone is 1,036) because Telegram hand-builds the whole UI (pickers, config,
  regenerate, status) that React gives the web channel for free. Every feature is built twice and
  drifts.
- **Good parts (keep):** the `X.py` (framework-free, unit-testable) / `X_runtime.py` (aiogram IO)
  split in Telegram; the tidy `SSEChannel`/`GoogleChatChannel` wiring; the explicit `registry.py`.
- **Fix direction:** same shape as 0.1/0.2 — split pull-transport vs side-effect delivery, make
  `deliver` return nothing and adapt to bytes only at the HTTP edge; delete `ChannelResponse` or
  actually use it; fix the stale docstrings.

## 1. Headline (automated hunt)

Pawrrtal's rot clusters into one dominant pattern — governance/observability and entire subsystems wired end-to-end but dead at one missing seam — plus a handful of real security/correctness bugs (path-jail bypass, symlink tree escape, multi-worker OAuth, plaintext OAuth password) and pervasive copy-paste of cross-cutting invariants across providers/channels/frontend. Three security findings and four silent-correctness findings are verified against source; the recon's headline claims hold up.

## 2. Ranked findings

### 🔴 #1 — Workspace path-jail bypassed via str.startswith() in markitdown_convert and send_message — sibling-prefixed workspace escape
- **Severity:** HIGH · **Category:** security · **Area:** `backend/app/tools`
- **Evidence:** markitdown_convert.py:49 `if not str(target).startswith(str(root))` (root NOT resolved first); send_message.py:84 `if not str(target).startswith(str(root.resolve()))`. Canonical guard at workspace_files.py:170 uses `target.is_relative_to(root_resolved)` with an in-source comment naming this exact sibling-prefix attack (`/data/workspaces/abc` vs `/data/workspaces/abcdef`). python_exec.py:69 correctly imports the canonical `_resolve_safe`; these two re-implement it weakly.
- **Why it's rot:** A path resolving to a sibling workspace whose root string shares a prefix (no separator) passes the startswith guard but would fail is_relative_to. The agent (or user code via the python tool's os.symlink) can read or exfiltrate files from another tenant's workspace. markitdown additionally omits root.resolve(), widening the gap.
- **Fix:** Replace both startswith checks with `target.is_relative_to(root_resolved)`; import the canonical `_resolve_safe` from workspace_files.py (already done by python_exec.py) and delete the two local copies.

### 🔴 #2 — GET /{workspace_id}/tree follows symlinks — host-filesystem enumeration outside the workspace
- **Severity:** HIGH · **Category:** security · **Area:** `backend/app/workspace/filesystem.py`
- **Evidence:** build_tree (filesystem.py:40-69) calls root.iterdir() and branches on `entry.is_dir()` (line 57) with no `entry.is_symlink()` guard; Path.is_dir() follows symlinks, so a symlink to /etc or /home is recursed into. The per-file read/write/delete routes are protected by has_symlink_parent/safe_child, but build_tree calls neither.
- **Why it's rot:** An agent or user who can create a symlink inside their workspace (agent tools, or python_exec os.symlink) can enumerate names and sizes of any path the process user can read, via the tree endpoint. Confirmed: no is_symlink check exists anywhere in build_tree.
- **Fix:** Add `if entry.is_symlink(): continue` before the is_dir branch in build_tree; audit serve_default_workspace_file for the same gap.

### 🔴 #3 — OAuth state stored in module-level in-process dict — every login fails under multi-worker uvicorn
- **Severity:** HIGH · **Category:** bug-risk · **Area:** `backend/app/infrastructure/auth/oauth/router.py`
- **Evidence:** router.py:37 `_state_cache: dict[str, float] = {}` with an in-source comment 'In a multi-process deployment swap this for Redis or a signed JWT'. _issue_state and _consume_state operate only on this dict. The Google flow (the primary web login) depends on it.
- **Why it's rot:** Under `uvicorn --workers N` (production default) the callback can hit a different worker than the one that issued the state, so _consume_state returns False and every login fails with 400 'Invalid OAuth state'. Invisible in single-worker local dev and tests — exactly where it would be caught.
- **Fix:** Round-trip a signed JWT state through redirect_uri (stateless), or move state to Redis/DB with TTL.

### 🔴 #4 — Scheduled/heartbeat turns silently drop ScheduledEvent.working_directory and run plugin-less, hook-less
- **Severity:** HIGH · **Category:** bug-risk · **Area:** `backend/app/infrastructure/event_bus/handlers.py`
- **Evidence:** working_directory is carried all the way into the event (types.py:116) and into _fire_event kwargs (scheduler.py:288 `working_directory: row.working_directory`), but _run_agent_turn(*, prompt, user_id) (handlers.py:361) takes only prompt+user_id, calls get_default_workspace(user_id) at line ~382, and never reads it; handle_scheduled (handlers.py:113-119) never forwards it. Separately, build_agent_tools is called with no workspace_id/conversation_id, so _build_plugin_tools (tools.py:322) short-circuits and build_pre_turn_hooks() is never called (it IS called in chat/router.py and telegram/bot.py).
- **Why it's rot:** Two silent capability gaps on the exact automated turns users configure to run unattended: (a) a reminder scheduled against a specific working_directory runs against the user's default workspace — wrong files, wrong HEARTBEAT.md/AGENTS.md; (b) Notion/image plugins and active_recall context injection are absent for every scheduled run.
- **Fix:** Thread working_directory through _run_and_publish into _run_agent_turn and skip get_default_workspace when supplied; resolve workspace_id and pass it to build_agent_tools; call build_pre_turn_hooks() before the stream.

### 🔴 #5 — One-shot scheduled jobs never deactivated — is_active stays True and rows re-hydrated on every restart
- **Severity:** HIGH · **Category:** bug-risk · **Area:** `backend/app/agents/scheduling/scheduler.py`
- **Evidence:** _mark_fired (scheduler.py:329) sets last_status='fired' but the `is_active = False` lines are commented out; _hydrate_active_jobs (scheduler.py:218-267) re-registers ALL is_active=True rows and the entire past-fire_at missed-job tombstone block is commented out, re-registering past one-shots as `DateTrigger(run_date=row.fire_at)`. APScheduler is pinned >=3.10 (pyproject.toml:13); no misfire_grace_time/coalesce is configured on add_job.
- **Why it's rot:** One-shot reminders are never tombstoned, so DB state is permanently wrong (is_active=True forever) and past-date DateTriggers are re-created on every restart. Whether they re-fire immediately (vs. drop as misfire) is APScheduler-config-dependent, but the state-corruption and re-registration are unambiguous; the also-commented failed/missed status writes (handlers.py:153-188) mean failures are invisible too.
- **Fix:** Uncomment/implement is_active=False in _mark_fired for fire_at rows; implement the missed-job skip/tombstone in _hydrate_active_jobs; set an explicit misfire_grace_time and coalesce on add_job.

### 🔴 #6 — AuditLogger has zero production call sites — the entire audit trail is hollow, and permission_audit_sink is never populated
- **Severity:** HIGH · **Category:** dead-code · **Area:** `backend/app/governance`
- **Evidence:** `grep -rln AuditLogger backend/app` excluding governance/ returns nothing; the only near-match, loop.py:87 `_execute_and_log_tool_call`, uses stdlib logging, not AuditLogger. permission_audit_sink is declared on AgentLoopConfig (types.py:489) and called at loop.py:828 only when non-None, but all five provider construction sites omit it (sole non-test reference is test scaffolding). audit_log_enabled setting, the AuditEvent ORM table, CRUD, and dashboard API all exist.
- **Why it's rot:** Every tool call, permission denial, file access and auth event produces zero audit rows; the dashboard is empty regardless of traffic. Security violations from the permission gate are swallowed with no log above INFO. A designed-and-wired-to-nothing governance feature.
- **Fix:** Construct an AuditLogger closure where PermissionContext is built (turn_runner/chat router), pass it as permission_audit_sink to each AgentLoopConfig, and call log_tool_call from _execute_and_log_tool_call.

## 3. Themes

### Designed-and-wired-but-dead-at-one-seam subsystems _(n=8)_
The dominant pattern: a feature has its ORM table, config flag, CRUD, schema, CLI/dashboard rendering, and even its event subscriber — but the single connecting call is missing or commented out, so it silently does nothing. AuditLogger (no caller), permission_audit_sink (never passed), CostBudgetMiddleware (never mounted), WebhookEvent (emitter deleted, handler still subscribed), TurnStarted/CompletedEvent (published, zero subscribers), scheduled-job failed/missed status (write path comment-locked), scheduled working_directory (carried then ignored). All look live from settings and from the DB schema but produce no effect.

### Cross-cutting invariants copy-pasted instead of shared _(n=10)_
Load-bearing invariants are re-implemented per-call-site and have already diverged. Path-resolution _resolve_safe (3 copies, only 1 correct — security), _is_sensitive_key redaction (2 divergent rule sets — security), history→AgentMessage and StreamFn-error blocks (4 providers each), PROVIDER='telegram' (9 files), TelegramSenderLike Protocol (3 pickers), picker turn-context resolution (5 copies), permission-gate closures (chat vs telegram), three frontend hooks with a diverged event name, and the chat/telegram history-window constant. Bug fixes must be applied N times and silently aren't.

### Workspace-path / filesystem trust boundary is inconsistently enforced _(n=3)_
The workspace jail is correct in exactly one place (workspace_files._resolve_safe with is_relative_to) and weaker or absent elsewhere: startswith() in two tools (sibling-prefix escape), no symlink guard in build_tree (host enumeration). The correct implementation even documents the attack it prevents, but the siblings never adopted it.

### Auth/OAuth path is structurally unsound for production _(n=4)_
In-process state dict (fails under --workers), raw token stored as hashed_password (corrupts fastapi-users reset flow), Apple callback permanently 501 while apple_start redirects into it, and the whole OAuth router has zero tests.
