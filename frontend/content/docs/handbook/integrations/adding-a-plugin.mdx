# Adding a Plugin

Use the Paw CLI when creating a workspace CLI plugin:

```bash
paw plugins scaffold local_search --tool-name search_notes --workspace-root ~/paw --json
paw plugins validate ~/paw/.agent/plugins/local_search --json
paw plugins reload --workspace-root ~/paw --json
```

Plugins are manifest-backed bundles discovered from three roots:

```text
backend/plugins/<plugin_id>/                 # bundled with Pawrrtal
$PAWRRTAL_HOME/plugins/<plugin_id>/          # global extra plugins
<workspace>/.agent/plugins/<plugin_id>/      # workspace plugins
```

Every plugin has a `plugin.json`. The runtime loads manifests through
`app.plugins`, persists enablement in `.agent/plugin-state/*.json` for a
workspace, and hot-reloads the next time tools or turn context providers are
composed.

## Minimal CLI Tool Plugin

Workspace plugins can safely contribute CLI tools. Put this in
`<workspace>/.agent/plugins/example_cli/plugin.json`:

```json
{
  "schema_version": 1,
  "id": "example_cli",
  "name": "Example CLI",
  "description": "Run a workspace-local command as an agent tool.",
  "version": "1.0.0",
  "permissions": ["subprocess"],
  "capabilities": [
    {
      "type": "cli_tool",
      "id": "example_echo",
      "tool_name": "example_echo",
      "title": "Example Echo",
      "description": "Echo arguments through a workspace CLI command.",
      "exposure": "direct_and_catalog",
      "entrypoint": ["python3", "-c", "import json, sys; print(json.dumps(sys.argv[1:]))"],
      "cwd": "workspace",
      "args_schema": {
        "type": "object",
        "properties": {
          "args": {
            "type": "array",
            "items": {"type": "string"}
          }
        },
        "required": [],
        "additionalProperties": false
      }
    }
  ]
}
```

`exposure: "direct_and_catalog"` makes the tool available to the agent and
searchable in the plugin capability catalog. `exposure: "catalog"` keeps it
searchable without adding it to the model's direct tool list.

## Env Settings

Declare env keys in a `settings` capability. Workspace and user-workspace keys
with `overridable: true` are automatically accepted by the workspace env API, so
new plugins do not need frontend/backend allowlist edits.

```json
{
  "type": "settings",
  "id": "example_settings",
  "title": "Example Settings",
  "description": "Workspace-scoped Example service credentials.",
  "env": [
    {
      "name": "EXAMPLE_API_KEY",
      "required": true,
      "scope": "workspace",
      "overridable": true,
      "gateway_fallback": false,
      "secret": true,
      "label": "Example API Key",
      "description": "Token used by the Example CLI."
    }
  ]
}
```

Required env keys gate plugin activation. A plugin with missing required env is
inspectable, but its direct capabilities are not exposed until configured.
Optional env keys still appear in workspace settings but do not block
activation. The bundled `github_issues` plugin uses this shape:
`GITHUB_TOKEN` is required to expose `report_issue`, while
`GITHUB_ISSUES_REPO` can override the gateway default per workspace.

## Bundled Python Tools

`python_tool` capabilities are bundled-only. They import first-party Python
factories and receive `ToolContext`, so they must live in `backend/plugins` and
point at code under `backend/app/plugins`. Workspace and global plugins should
use `cli_tool` unless the host grows a narrower trust boundary for them.

The bundled Tasks plugin is the template:

```json
{
  "type": "python_tool",
  "id": "add_task",
  "tool_name": "add_task",
  "title": "Add Task",
  "description": "Append a new task to the workspace TASKS.md file.",
  "slots": ["tasks"],
  "exposure": "direct_and_catalog",
  "permissions": ["filesystem_write"],
  "entrypoint": "app.plugins.tasks.tools:make_add_task_tool"
}
```

The factory signature is:

```python
def make_add_task_tool(ctx: ToolContext) -> AgentTool: ...
```

`ToolContext` carries the active `workspace_id`, `workspace_root`, `user_id`,
and optional turn metadata such as `conversation_id`, `model_id`, and
`send_fn`. Use those fields instead of importing request/session state inside a
plugin factory.

The returned `AgentTool.name` must match `tool_name`; mismatches are logged and
omitted from the direct tool list.

High-risk tools should be off by default. The bundled `python_shell` plugin
declares the `python` tool as a `python_tool`, sets `enabled_by_default: false`,
and relies on workspace plugin state for explicit opt-in. Its timeout and output
cap remain host settings because they protect the FastAPI worker, not a single
workspace.

The bundled `reminders` plugin is the scheduler-backed Python-tool template. It
keeps `reminder_schedule`, `reminder_list`, and `reminder_cancel` out of the
kernel tool builder while still preserving the active conversation id when a
reminder should target the current chat.

The bundled `github_issues` plugin is the env-gated integration template. It
declares `report_issue` as a `python_tool` and contributes settings metadata so
each workspace can bring its own GitHub token and issue target.

The bundled `document_conversion` plugin is the local-file adapter template. It
declares `convert_to_markdown` as a `python_tool`, keeps MarkItDown scoped to
the workspace root, and requires only `filesystem_read`.

The bundled `image_generation` plugin is the opt-in costly tool template. It is
disabled by default, requires `OPENAI_CODEX_OAUTH_TOKEN`, and only exposes
`generate_image` after the workspace enables the plugin and configures the
credential.

The bundled `exa_search` plugin is the first web-search slot implementation. It
is disabled by default, requires `EXA_API_KEY`, and declares `web_search` as a
non-exclusive slot so other search providers can be installed beside it.

## Trusted Runtime Capabilities

Runtime capabilities such as `turn_context_provider`, `provider`, `channel`,
`router`, and `scheduler` are trusted adapters. They are allowed from bundled
and global plugin roots, not workspace roots.

Active Recall is the first bundled `turn_context_provider`:

```json
{
  "type": "turn_context_provider",
  "id": "active_recall",
  "title": "Active Recall",
  "description": "Find relevant prior context before the main model answers the user.",
  "slots": ["turn_context"],
  "entrypoint": "app.plugins.active_recall.recall_agent:run_active_recall",
  "order": 100,
  "budget": {
    "timeout_seconds": 10,
    "failure_mode": "omit"
  }
}
```

The provider receives `TurnContextProviderContext` and returns either a short
context string or `None`. The turn orchestrator owns prompt assembly and wraps
all provider output under `# TURN CONTEXT`.

`backend/plugins/core_channels/plugin.json` is the bundled channel example. A
`channel` capability maps a surface name (`web`, `electron`, `telegram`,
`google_chat`) to a trusted factory that returns a `Channel` adapter. The
channel registry composes those capabilities through the plugin host and keeps
only `web` as a kernel fallback.

Provider plugins use `provider` capabilities. Their model IDs are dynamic and
do not need a `Host` or `Vendor` enum entry. The canonical shape is:

```text
plugin_id:vendor/model
```

The provider entrypoint is a trusted factory that receives
`ProviderFactoryContext` and returns an `AILLM`. Active provider-plugin models
are appended to `/api/v1/models` and can be selected by the normal chat API.

## Enablement And Hot Reload

Enablement lives outside the manifest:

```text
<workspace>/.agent/plugin-state/<plugin_id>.json
```

Missing state defaults to enabled only for bundled plugins with
`enabled_by_default: true`. Workspace and global plugins must be explicitly
enabled by plugin state or the Paw CLI. The host reloads from disk when composing
plugin tools or turn context providers, so adding, removing, enabling, disabling,
or editing a plugin is visible on the next turn.

## Agent Discovery

Every agent turn gets a core `search_plugin_capabilities` tool. Use it before
assuming which optional plugin, provider, channel, or slot candidate exists:

```json
{
  "slot": "web_search",
  "include_unavailable": true
}
```

The tool returns capability rows with composite keys (`plugin_id/capability_id`),
types, slots, state, plugin load status, missing env names, and whether a row is
preferred for the requested slot. This is intentionally separate from direct
tool invocation: users can install several candidates for a slot and the agent
can inspect all of them.

## Tests

Use these focused tests as examples:

- `backend/tests/test_plugin_tools.py` covers direct CLI tool exposure and hot reload.
- `backend/tests/test_plugin_tools.py` covers bundled Python tools, env-gated issue reporting, reminder context, and the non-bundled Python trust boundary.
- `backend/tests/test_plugin_catalog_tool.py` covers the agent-facing capability search tool.
- `backend/tests/test_turn_context_plugins.py` covers Active Recall as a manifest-backed context provider.
- `backend/tests/test_workspace_env_api.py` covers plugin-declared workspace env overrides.

At minimum, verify:

```bash
cd backend && uv run pytest tests/test_plugin_tools.py tests/test_plugin_catalog_tool.py tests/test_turn_context_plugins.py tests/test_workspace_env_api.py
```
