# xAI (Grok)

Grok is a Pawrrtal chat provider on the same `AILLM` seam as Claude and
Gemini. The provider sits on xAI's official `xai-sdk` (gRPC, Python
3.10+); the design rationale and migration history are in
[`2026-05-18-xai-grok-sdk-swap`](../decisions/2026-05-18-xai-grok-sdk-swap).
This page is the operator-facing reference — model catalog entry, env
keys, and the small surface a developer typically touches.

## At a glance

| Field | Value |
|---|---|
| Vendor | `xai` |
| Host | `xai` (native gRPC SDK) |
| Canonical model ID | `xai:xai/grok-4.3` |
| Model slug | `grok-4.3` |
| Context window | 1M tokens |
| Price (input / output) | $1.25 / $2.50 per 1M tokens |
| SDK | [`xai-sdk>=1.12.0`](https://github.com/xai-org/xai-sdk-python) |
| Provider class | `backend/app/core/providers/xai_provider.py:XaiLLM` |
| Env key | `XAI_API_KEY` (workspace-overridable) |

The canonical wire form is `xai:xai/grok-4.3`; the bare
`xai/grok-4.3` resolves to the same triple via `CANONICAL_HOST`
(see the [canonical model-ID
ADR](../decisions/2026-05-14-model-id-canonical-format-and-backend-catalog)).

## Catalog entry

Grok 4.3 lives in `backend/app/core/providers/catalog.py` and surfaces
through `GET /api/v1/models` without any router or schema change:

```python
ModelEntry(
    host=Host.xai,
    vendor=Vendor.xai,
    model="grok-4.3",
    display_name="Grok 4.3",
    short_name="Grok 4.3",
    description="xAI's frontier 1M-context model",
    is_default=False,
    cost_per_mtok_in_usd=_GROK_4_3_IN_USD,   # 1.25
    cost_per_mtok_out_usd=_GROK_4_3_OUT_USD, # 2.50
)
```

Adding another Grok variant is a single catalog edit — no provider,
factory, or schema change.

## API keys

`XAI_API_KEY` follows Pawrrtal's standard env-key resolution. The chat
path calls `resolve_api_key(workspace_id, "XAI_API_KEY")`, which checks
the workspace-encrypted store first and falls back to
`settings.xai_api_key` from the gateway env.

To configure per-workspace, the user pastes the key into Settings →
Workspace Env (this works because `XAI_API_KEY` is on
`OVERRIDABLE_KEYS` in `backend/app/core/keys.py` and mirrored in the
frontend's `WORKSPACE_ENV_KEY_IDS` + `KEY_METAS` for the settings row).

For local dev without a configured workspace, set `XAI_API_KEY` in the
backend `.env`. The STT path (`backend/app/api/stt.py`) honours the
same key.

## Reasoning effort

Pawrrtal's chat composer exposes a 4-bucket UI knob:
`low | medium | high | extra-high`. Grok 4.3 only accepts two levels
(`EFFORT_LOW` / `EFFORT_HIGH`) and 400s on anything else, including
`EFFORT_MEDIUM` and `EFFORT_NONE`. `_map_reasoning_effort` collapses
the 4 buckets onto 2:

| UI selection | Wire value |
|---|---|
| `low`, `medium` | `chat_pb2.ReasoningEffort.EFFORT_LOW` |
| `high`, `extra-high` | `chat_pb2.ReasoningEffort.EFFORT_HIGH` |
| `None` | field omitted, server picks the model default |

The model's chain-of-thought streams back as `reasoning_content`
deltas. The provider surfaces those as `LLMThinkingDeltaEvent` so the
existing thinking pane (wired for Gemini per issue #98) lights up for
Grok with no frontend change.

## How the provider is wired

`XaiLLM.stream` builds a per-request `StreamFn` that opens a fresh
`xai_sdk.AsyncClient` inside an `async with` block so the gRPC channel
closes deterministically after every request. The closure captures the
assembled system prompt (SOUL.md + AGENTS.md + skills, rendered as
`ROLE_DEVELOPER`), the optional `reasoning_effort` enum, and a
per-turn `UsageAccumulator` that the cost ledger consumes.

```
┌──────────────────────────────────────────────────────┐
│ api/chat.py                                          │
│   ↓ resolve_llm("xai:xai/grok-4.3", workspace_id)    │
│ providers/factory.py                                 │
│   ↓ HOST_TO_PROVIDER[Host.xai] → XaiLLM              │
│ providers/xai_provider.py: XaiLLM.stream             │
│   ↓ make_xai_stream_fn(model, ws, system_prompt,…)   │
│ providers/_xai_messages.py                           │
│   ↓ build_xai_messages / build_xai_tools (protos)    │
│ xai_sdk.AsyncClient.chat.create(...).stream()        │
│   ↓ Chunk / Response                                 │
│ providers/_xai_stream.py                             │
│   ↓ deltas / tool_calls / usage → LLMEvent shapes    │
│ agent_loop → providers/_xai_events.py                │
│   ↓ AgentEvent → StreamEvent (text / thinking /      │
│     tool_result / done / usage)                      │
└──────────────────────────────────────────────────────┘
```

Files (all under `backend/app/core/providers/`):

- `xai_provider.py` — the `XaiLLM` class + `make_xai_stream_fn`.
- `_xai_messages.py` — typed `chat_pb2.Message` / `chat_pb2.Tool`
  builders. Assistant messages with `tool_calls` build the proto
  directly because `xai_sdk.chat.assistant()` only exposes
  text / image / file content.
- `_xai_stream.py` — `Chunk` → `LLMEvent` translation, tool-call
  extraction off the accumulated `Response`, `UsageAccumulator`
  driven by `Response.cost_usd` + `Response.usage.{prompt,completion}_tokens`.
- `_xai_events.py` — `AgentEvent` → `StreamEvent`.

The split is enforced by the project's 500-line file-line budget
(`scripts/check-file-lines.mjs`).

## What's intentionally not done yet

- **Multimodal image inputs.** `XaiLLM.stream` accepts the `images`
  argument for protocol parity but logs-and-drops a non-empty list.
  Bridging to `xai_sdk.chat.image` is a small follow-up.
- **Responses API.** The provider currently uses the chat-completions
  surface on `xai-sdk`. The Responses API (`web_search` server-tool,
  deferred completion, multi-turn `previous_response_id`) is the next
  layer; no wire-library swap needed.
- **Live Search.** xAI's built-in Live Search is no longer wired —
  Pawrrtal's canonical web tool is `exa_search`. If xAI ships a
  replacement search surface on the Responses API, the knob will flow
  through `SearchParameters`.

## Testing

Provider behaviour is covered by `backend/tests/test_xai_provider_translation.py`
(39 tests covering proto-message construction, chunk/delta extraction,
tool-call extraction off the accumulated response, finish-reason
translation, usage capture, error fall-through, and the
`ROLE_DEVELOPER` system-prompt placement) and the scripted-trajectory
tests in `test_xai_stream_fn.py`. Both follow the project's
[ScriptedStreamFn pattern](../../../../.claude/rules/testing/agent-loop-testing-philosophy.md).
