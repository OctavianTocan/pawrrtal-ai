# Capability Plugin Platform Implementation Plan

This plan implements the
[Capability Plugin Platform ADR](../decisions/2026-06-06-plugin-system-v2).
It is intentionally concrete: an engineer or agent should be able to execute
the slices in order without deciding the architecture again.

## Goal

Thin the Pawrrtal kernel down to auth, users, workspaces, conversations,
message persistence, turn orchestration, audit/cost, and the plugin host.
Everything that is optional or integration-specific becomes a plugin:

- tools and CLI tools;
- agent profiles that define reusable delegated workers;
- turn context providers, turn observers, lifecycle tasks, and scheduled jobs;
- providers and model catalog rows;
- agent runtime adapters that own native session/thread/tool-event runtimes;
- channels and channel ingress/delivery;
- migration/import providers;
- plugin-owned routers;
- settings/env metadata;
- Paw-readable skills and authoring instructions.

No backward compatibility with the old Python plugin registry is required.
Data migrations are still required so existing local data remains coherent.

## Non-Negotiables

- The new module is deep: callers ask for active capabilities and do not know
  whether they came from bundled Python, global plugins, or workspace CLI
  plugins.
- The Paw CLI is part of the development process. Every implementation slice
  that touches plugins must use `paw plugins spec`, `paw plugins validate`,
  `paw plugins doctor`, `paw plugins graph`, and `paw plugins reload` once
  those commands exist.
- Runtime-created workspace plugins are supported in v1 for CLI tools,
  settings metadata, skills, and agent profiles.
- Plugin-declared env vars are dynamically overridable per workspace. Each
  user-owned workspace can carry its own values for provider, channel, tool,
  and CLI plugin secrets.
- Hot reload is supported without mutating in-use objects. Every request, chat
  turn, provider stream, channel delivery loop, and scheduled job runs against
  the registry snapshot it started with.
- Channel ingress uses stable kernel dispatch routes so runtime reload does
  not depend on mutating the FastAPI router table.
- Python entrypoint hot reload is development-only. Production reloads
  manifests, settings, skills, CLI tools, model catalog metadata, and
  enablement state; production Python code changes require process restart.
- Workspace-authored plugins are CLI-only until a separate trust/sandbox
  decision allows workspace Python imports.
- Commits are small and story-driven. Commit messages must explain the
  architectural intent, not just list touched files.

## Reference Validation Findings

The plan was checked against local reference implementations and official
Claude Code subagent documentation before execution. The point is not to clone
another system; it is to copy the contracts that keep large plugin surfaces
maintainable.

| Reference | Validated pattern | Pawrrtal plan adjustment |
|---|---|---|
| Codex plugin store | Strict manifests, install/cache/data separation, enablement toggles, load outcomes, capability summaries, and deferred dynamic tools. | Keep definition, state, and runtime outcome separate. Add `exposure: "catalog"` as the Pawrrtal equivalent of deferred tools, and require structured load outcomes for failed/misconfigured plugins. |
| Codex extension API | Typed contributor registry for prompts, tools, lifecycle hooks, config, approval review, and tool lifecycle events. | Do not make a single generic plugin callback bag. Registry snapshots expose typed contribution indexes, and lifecycle/tool events have their own contracts. |
| OpenClaw plugins | Manifest-only discovery before executing runtime code, capability ownership contracts, provider/channel/tool shapes, plugin status/inspect, secret contracts, workspace-aware runtime state, and reload snapshots. | Add manifest-only control-plane metadata, explicit plugin shapes, status/inspect diagnostics, secret input contracts, shared channel message tool behavior, and immutable runtime snapshots. |
| Pi embedded runtime | Agent SDK embedded as a runtime layer with injected tools, event subscription, session persistence, auth rotation, and schema adaptation. | Add `agent_runtime` as a first-class adapter for Codex/Pi-like runtimes instead of forcing them into plain provider code. Do not expose runtimes as generic searchable capabilities. |
| Claude Code subagents | Named agent definitions carry prompt, tool access, model preference, scope, and delegation description while execution happens in a separate agent context. Plugin subagents are profile data with restricted fields, not runtime implementations. | Add `agent_profile` contributions for reusable personas and `delegated_agent_jobs` for executions. Plugins may ship profiles, but Pawrrtal owns job lifecycle, permissions, events, and persistence. |
| Hermes migration plugin | Migration is a provider capability with `detect`, `plan`, and `apply`, conflict handling, reports, and manual-review items. | Add `migration_provider` so Notion imports, Hermes/Claude imports, and future workspace migration flows are plugins rather than one-off scripts. |

Architecture guardrails copied from those references:

- Plugin discovery and config validation must work from manifest metadata
  without executing plugin runtime code.
- Runtime loading is a separate phase and produces a typed `LoadOutcome`, not
  a boolean.
- Runtime-created tools may be searchable and invokable without being directly
  injected into every model turn.
- Provider, channel, tool, turn-lifecycle, migration, and agent-runtime
  contributions are typed contributors with narrow contracts.
- Subagents are profile definitions plus delegated runtime jobs, not a plugin
  implementation category.
- Channels use a shared core message tool and plugin-owned action discovery
  instead of adding separate model tools for every channel action.
- Secret and env surfaces are declared by plugin manifests, resolved per
  workspace, and marked inactive when a plugin is disabled or no longer
  loadable.
- Cached discovery snapshots are explicit products of a flow. Hidden global
  manifest caches and wall-clock TTLs are not part of plugin loading.

## Plugin Sources And Precedence

The host discovers plugin definitions from three source roots:

| Source | Path | Trust | Hot reload |
|---|---|---|---|
| Bundled | `backend/plugins/<id>/plugin.json` | first-party | manifests yes, Python code dev-only |
| Global extra | `$PAWRRTAL_HOME/plugins/<id>/plugin.json` | local operator | manifests/CLI/settings/skills yes |
| Workspace | `<workspace_root>/.agent/plugins/<id>/plugin.json` | workspace user/Paw | CLI/settings/skills yes |

`PAWRRTAL_HOME` is resolved through the existing settings/config path that
backs local Pawrrtal state. If no explicit home exists, use the existing
Pawrrtal state directory convention, not the repo root.

Resolution rules:

- Plugin ids are lowercase `[a-z][a-z0-9_]{1,62}`.
- Duplicate ids fail validation unless the later plugin explicitly declares
  `overrides: "<existing_id>"`.
- Precedence is workspace over global over bundled, but only with explicit
  `overrides`.
- A plugin may depend on another plugin by id and minimum schema version.
- Dependency cycles fail validation.
- A disabled dependency disables dependents for that workspace.
- A failed plugin never prevents kernel boot; it appears as `failed` in the
  registry and management APIs.

## State Model

Definition files are not the same as enablement state.

Per-workspace state for every plugin id lives at:

```text
<workspace_root>/.agent/plugin-state/<plugin_id>.json
```

System/global state for process-level checks lives at:

```text
$PAWRRTAL_HOME/plugin-state/<plugin_id>.json
```

State file shape:

```json
{
  "enabled": true,
  "capabilities": {
    "notion_cli": {
      "enabled": true
    }
  },
  "slot_preferences": {
    "web_search": [
      "exa_search/web_search",
      "brave_search/web_search"
    ]
  },
  "validated_fingerprint": "sha256...",
  "validated_at": "2026-06-06T18:00:00Z",
  "last_validation": {
    "passed": true,
    "checks": [
      {
        "name": "manifest",
        "status": "passed",
        "message": "Manifest is valid."
      }
    ]
  },
  "failure_reason": null,
  "doctor": {
    "status": "ok",
    "checked_at": "2026-06-06T18:01:00Z"
  }
}
```

State rules:

- Writes are atomic: write temp file, fsync, rename.
- A `.lock` file protects concurrent enable/disable/validate/reload.
- State is deliberately outside `.agent/plugins/<plugin_id>/` so replacing,
  regenerating, or deleting a plugin definition does not silently delete
  enablement, validation, doctor, or slot-preference history.
- `enabled` is the plugin-level switch. When it is false, none of the plugin's
  capabilities are available to the Paw or direct provider tool lists.
- `capabilities.<capability_id>.enabled` is optional per-capability gating for
  cases where a user wants one plugin installed but only some capabilities
  active.
- `slot_preferences` stores ordered workspace preferences. It never disables
  non-preferred capabilities; it only changes result ordering and default
  selection.
- `validated_fingerprint` covers manifest, declared local files, validation
  commands, and CLI entrypoints. It does not include mutable state files.
- If the current fingerprint differs from `validated_fingerprint`, the plugin
  state becomes `needs_validation` and its runtime capabilities are inactive.
- Bundled default plugins may declare `enabled_by_default: true`. A workspace
  state file may still disable them.
- Missing state for a bundled default means enabled if validation passes.
- Missing state for global/workspace plugins means available but disabled.

## Manifest Contract

The top-level manifest is a Pydantic model in
`backend/app/plugins/manifest.py`.

```json
{
  "schema_version": 1,
  "id": "notion",
  "name": "Notion",
  "description": "Run Notion workspace operations through the ntn CLI.",
  "version": "1.0.0",
  "enabled_by_default": false,
  "overrides": null,
  "depends_on": [],
  "permissions": ["subprocess", "secrets"],
  "env": [],
  "capabilities": [],
  "validation": {
    "commands": []
  }
}
```

Common validation:

- `name` is Title Case and user-facing.
- `description` is one sentence and at least 20 characters.
- `permissions` must be declared before a capability can use that operation.
- `capabilities` must be non-empty.
- Every capability may declare `title`, `description`, `tags`, `intents`,
  `slots`, `priority`, and `exposure`.
- Capability ids are unique after override resolution.
- Router prefixes, channel surfaces, tool names, provider ids, scheduler ids,
  and env keys are globally unique within a registry snapshot.
- Unknown fields fail validation. This keeps manifests intentional.
- Env keys declared by plugins automatically become valid workspace env keys.
  Do not keep a parallel static allowlist in backend or frontend code.
- Control-plane discovery must be possible without importing plugin runtime
  code. Manifest fields own static identity, config schema, setup hints,
  capability ownership, env contracts, activation hints, and catalog metadata.
- Runtime code owns behavior only after the plugin is enabled, validated,
  configured for the active workspace, and selected by the host.
- Every plugin load produces a structured outcome: `active`, `disabled`,
  `needs_validation`, `misconfigured`, `failed`, or `blocked_by_dependency`.
  A failed plugin remains inspectable and never disappears from diagnostics.

## Env Override Contract

Plugin env declarations are the source of truth for workspace-overridable
secrets and settings.

Env spec shape:

```json
{
  "name": "NOTION_API_KEY",
  "inject_as": "NOTION_API_TOKEN",
  "required": true,
  "scope": "workspace",
  "overridable": true,
  "gateway_fallback": false,
  "secret": true,
  "label": "Notion API Key",
  "description": "Token used by the Notion CLI.",
  "help_url": "https://www.notion.so/profile/integrations"
}
```

Scopes:

| Scope | Meaning |
|---|---|
| `workspace` | Value lives in the encrypted env file for the active workspace. This is the v1 default and gives each user-owned workspace its own value. |
| `gateway` | Value comes only from process/global settings. Use for operator-owned deployment config. |
| `user_workspace` | Reserved for future shared workspaces where two users need different values inside the same workspace. In v1 it resolves the same as `workspace` because workspaces are user-owned. |

Resolution order:

1. Active workspace encrypted env override.
2. Gateway/global setting, only when `gateway_fallback` is true and the env
   spec maps to a settings attribute.
3. Missing.

Rules:

- A plugin that declares `required: true` is active only when resolution returns
  a non-empty value for the active workspace.
- The same plugin can be enabled and configured in one workspace and
  misconfigured in another.
- Empty strings clear the workspace override and expose the gateway fallback
  when one is allowed.
- Secret values are never returned by plugin APIs, Paw CLI JSON, or frontend
  settings responses.
- Plugin subprocesses receive only declared env keys, renamed through
  `inject_as` when present.
- Provider, channel, tool, scheduler, and validation commands all use the same
  resolver. No caller may read `settings.X` directly for a plugin-declared
  env key.
- Replacing `OVERRIDABLE_KEYS` is part of the migration: the allowlist becomes
  `active_registry.settings.env_keys` plus kernel-owned non-plugin keys.
- Plugin config schemas may declare `secret_inputs` paths. Secret scanners and
  runtime materialization walk only those manifest-declared paths, mark them
  inactive when the plugin is disabled or stale, and never resolve secrets for
  plugins that are not loadable in the active workspace.
- Inline secret refs and env refs are normalized through the same resolver as
  direct env keys. Plain strings stay plain strings unless the manifest says
  the path is a secret input.

## Capability Contracts

### Tool Capabilities

`cli_tool`:

```json
{
  "type": "cli_tool",
  "id": "notion_cli",
  "tool_name": "notion_cli",
  "title": "Notion CLI",
  "description": "Run ntn with an argument list.",
  "intents": ["notion.workspace"],
  "slots": ["workspace_knowledge"],
  "exposure": "direct_and_catalog",
  "entrypoint": ["ntn"],
  "cwd": "plugin",
  "timeout_seconds": 30,
  "output_cap_bytes": 32000,
  "args_schema": {
    "type": "object",
    "properties": {
      "args": {"type": "array", "items": {"type": "string"}},
      "stdin": {"type": "string"}
    },
    "required": ["args"]
  }
}
```

`python_tool`:

```json
{
  "type": "python_tool",
  "id": "image_generate",
  "tool_name": "image_generate",
  "entrypoint": "app.plugins.openai_codex_image_gen:make_tool"
}
```

Rules:

- Workspace plugins may only use `cli_tool`.
- CLI execution always uses argv lists, never a shell.
- CLI tools return a JSON envelope containing `stdout`, `stderr`,
  `returncode`, and optional `error`.
- Tool ordering remains: core tools, plugin tools, external MCP tools.
- `exposure: "direct"` exposes the tool directly in normal provider tool
  lists.
- `exposure: "catalog"` hides the tool from normal provider tool lists but
  makes it discoverable and invokable through the capability catalog tools.
- `exposure: "direct_and_catalog"` does both.
- `catalog` exposure is the default for runtime-created or high-cardinality
  tools. The Paw can find these through `capabilities_search` and invoke them
  explicitly without bloating every turn's direct tool list.
- Tool responses are decoded through a narrow host-owned response envelope.
  Invalid plugin output becomes a failed tool result with a diagnostic message,
  not arbitrary model-visible text.

## Capability Catalog, Search, And Slots

The capability catalog is a kernel-owned module. It is not optional and is not
implemented as a plugin. Its job is to let the Paw discover and use the active
runtime surface safely.

Core catalog tools always available to the Paw:

| Tool | Purpose |
|---|---|
| `capabilities_search` | Search effectively enabled capabilities by query, type, intent, slot, tag, plugin id, or required permission. |
| `capability_describe` | Return one capability's manifest metadata, JSON schema, permissions, env/config state, examples, and invocation mode. |
| `capability_invoke` | Invoke an enabled invokable capability by `(plugin_id, capability_id, input)` through the plugin host. |

`capabilities_search` defaults to enabled capabilities only. It may include
disabled/failed/misconfigured capabilities only when called with
`include_unavailable=true` by a diagnostic tool path.

Capability effective state:

```text
effective_enabled =
  plugin_state.enabled
  and capability_state.enabled
  and dependencies_enabled
  and required_env_configured_for_workspace
  and validation_fingerprint_current
  and lifecycle_ready
```

Search result shape:

```json
{
  "plugin_id": "exa_search",
  "capability_id": "web_search",
  "type": "cli_tool",
  "title": "Exa Web Search",
  "description": "Search the public web with Exa.",
  "intents": ["web.search"],
  "slots": ["web_search"],
  "state": "enabled",
  "preferred": true,
  "priority": 100,
  "requires_confirmation": false,
  "permissions": ["network"],
  "input_schema": {"type": "object"},
  "examples": []
}
```

### Capability Invoke Contract

`capability_invoke` is the only generic runtime path for catalog-only or
deferred capabilities. It accepts:

```json
{
  "plugin_id": "exa_search",
  "capability_id": "web_search",
  "input": {"query": "Pawrrtal plugin examples"},
  "idempotency_key": "optional-stable-key",
  "confirmation": null
}
```

Invocation order:

1. Capture the current registry snapshot.
2. Resolve `(plugin_id, capability_id)` inside that snapshot.
3. Refuse missing, disabled, misconfigured, failed, dependency-blocked, stale
   fingerprint, or non-invokable capabilities before input validation.
4. Validate `input` against the capability JSON Schema. Unknown properties are
   rejected unless the schema explicitly allows them.
5. Evaluate declared permissions and risk.
6. Require explicit confirmation for `external_write`, `destructive`,
   `costly`, `publishing`, `credentialed_mutation`, and any capability that
   declares `requires_confirmation: true`.
7. Execute through the typed adapter with the captured snapshot and the active
   workspace env resolver.
8. Decode the adapter result into the host-owned output envelope.
9. Write `plugin_operation_logs` before returning.

Capabilities may declare:

```json
{
  "risk": "external_read",
  "requires_confirmation": false,
  "output_schema": {"type": "object"}
}
```

Allowed risk values:

| Risk | Confirmation |
|---|---|
| `local_read` | no |
| `external_read` | no, unless it has cost/privacy implications |
| `external_write` | yes |
| `destructive` | yes |
| `costly` | yes |
| `publishing` | yes |
| `credentialed_mutation` | yes |

Generic output envelope:

```json
{
  "success": true,
  "content_items": [
    {"type": "text", "text": "Search completed."}
  ],
  "data": {"results": []},
  "error": null,
  "operation_log_id": "uuid"
}
```

Rules:

- `content_items` are the only model-visible text/image output.
- `data` is structured machine output for UI, audit, or follow-up tools.
- Invalid adapter output becomes `success: false` with a diagnostic error.
- CLI capabilities adapt `stdout`, `stderr`, and `returncode` into `data`; the
  model sees only a bounded summary unless the tool contract says stdout is the
  intended model-visible payload.
- Provider and agent-runtime adapters project `catalog` capabilities as
  deferred/searchable tools. They must not place catalog-only tools in the
  ordinary direct tool list for every turn.
- Runtime adapters that support native deferred tools, such as Codex-style
  dynamic tools, may preserve the provider-native deferred flag. Runtimes that
  do not support native deferred tools invoke through the kernel catalog tools.

Slots are many-to-many labels, not singleton locks.

Slot metadata shape:

```json
{
  "id": "web_search",
  "title": "Web Search",
  "mode": "multi",
  "selection": "prefer_ordered"
}
```

Rules:

- A capability can declare zero or more slots.
- Multiple enabled capabilities may satisfy the same slot.
- Workspace preferences may mark one or more preferred capabilities for a slot.
- Search results are sorted by preferred status, health, manifest priority,
  then plugin id/capability id for stability.
- The Paw may choose among enabled candidates when the task is low-risk.
- The Paw should ask or use an explicit user preference when the choice affects
  cost, external writes, privacy, or user-visible publishing.
- Kernel-required choices, such as the default chat provider or default local
  channel, use explicit preferences rather than singleton slot enforcement.
- Exclusive slots are allowed only when the slot manifest declares
  `mode: "exclusive"`. Selecting a new exclusive-slot owner must return
  warnings that name the previous owner and every capability disabled or
  deprioritized by the change. Web Search is not exclusive.

Example slot behavior:

- `web_search` can be satisfied by Exa, Tavily, Brave Search, a local search
  CLI, or several of them at once.
- The user can enable several web search plugins and set Exa as preferred.
- `capabilities_search(slot="web_search")` returns all enabled candidates with
  Exa first.
- If no preference exists, the Paw can choose based on query fit or ask the
  user when the tradeoff matters.

### Turn Lifecycle Capabilities

```json
{
  "type": "turn_context_provider",
  "id": "active_recall",
  "entrypoint": "app.plugins.active_recall:create_context_provider",
  "order": 100,
  "budget": {
    "max_chars": 600,
    "failure_mode": "omit"
  }
}
```

Rules:

- Turn context provider ordering is stable by `order`, then plugin id, then
  capability id.
- Turn context providers run against the same registry snapshot as the turn.
- Turn context providers return typed `ContextContribution` objects. The kernel,
  not the plugin, owns final prompt assembly, ordering, truncation, and source
  labels.
- Turn context provider failures are classified as `blocking` or
  `non_blocking` by manifest.
- Turn observers are separate capabilities that run after turn finalization for
  memory writes, analytics, dreaming, and other side effects that must not alter
  the current main turn.

### Provider Capabilities

```json
{
  "type": "provider",
  "id": "openai_codex",
  "entrypoint": "app.plugins.providers.openai_codex:create_provider",
  "auth": {
    "checks": ["OPENAI_CODEX_OAUTH_TOKEN", "codex_auth_file"]
  },
  "models": [
    {
      "id": "openai-codex:openai/gpt-5.5-codex",
      "name": "GPT-5.5 Codex",
      "sort_order": 300,
      "default": true,
      "reasoning": ["minimal", "low", "medium", "high"],
      "supports_images": true,
      "supports_tools": true
    }
  ],
  "lifecycle": {
    "shutdown": "app.plugins.providers.openai_codex:shutdown"
  }
}
```

Rules:

- Remove `Host` and `Vendor` enums. Model ids remain strings shaped as
  `host:vendor/model`, validated by regex and active catalog membership.
- Provider plugins construct `AILLM` instances through a kernel
  `ProviderFactoryContext`.
- Provider plugins own auth checks, labels, setup help, model metadata,
  cost metadata, cache lifecycle, and shutdown hooks.
- Provider auth checks resolve plugin-declared env keys through the active
  workspace. This lets each user-owned workspace use its own provider account
  or token.
- Duplicate model ids fail registry validation.
- Multiple default models fail validation unless only one default is active
  in the current workspace snapshot.
- Provider manifests must expose cheap catalog/setup/auth metadata without
  importing the full provider runtime. Live model discovery may call provider
  runtime only after the plugin is active for the workspace.
- If a provider delegates to an agent runtime that owns sessions, compaction,
  tool events, or native thread state, that runtime must be registered as an
  `agent_runtime` adapter. The provider may depend on the runtime adapter, but
  core must not embed the runtime protocol in provider dispatch.

### Agent Runtime Adapters

Agent runtime adapters wrap runtimes such as Codex, Pi, Claude Agent SDK, or
other native agent SDKs that own session files, tool-call event streams,
compaction, delegated agent jobs, or provider-specific thread state. They are
plugin contributions, but they are not generic catalog capabilities: the Paw can
inspect their health and dependencies, while providers and delegation tools use
them through explicit runtime interfaces.

```json
{
  "type": "agent_runtime",
  "id": "openai_codex_runtime",
  "entrypoint": "app.plugins.agent_runtimes.openai_codex:create_runtime",
  "sessions": {
    "state_table": "conversation_provider_state",
    "native_thread_key": "codex_thread_id"
  },
  "events": ["turn_start", "tool_start", "tool_finish", "turn_stop"],
  "tool_schema_adapter": "responses_api"
}
```

Rules:

- Agent runtime adapters expose a narrow but complete interface:
  - `prepare_turn(input)` builds provider-native prompt/session input from
    normalized messages, system fragments, skills, images, files, and channel
    context.
  - `run_turn(input)` starts a new runtime turn.
  - `resume_turn(input)` resumes a native session or thread.
  - `abort_turn(run_id)` cancels one active run.
  - `compact_session(session_ref)` compacts or summarizes native history.
  - `subscribe_events(run_id, sink)` streams normalized lifecycle events.
  - `classify_failure(error, transcript_tail)` returns terminal categories
    such as `auth`, `rate_limit`, `quota`, `timeout`, `context_overflow`,
    `tool_schema`, `provider_unavailable`, or `unknown`.
- Agent runtime adapters receive a snapshot of active tools, not mutable access to the
  plugin host.
- Agent runtime adapters adapt tool schemas and event streams at the boundary. Provider
  code should not contain Codex/Pi/Hermes-specific event translation.
- Agent runtime adapters normalize tool lifecycle into `tool_start`, `tool_progress`,
  `tool_finish`, and `tool_error`. Before/after tool-call policy hooks attach
  at this boundary so every runtime gets the same audit and approval behavior.
- Agent runtime adapters receive auth/profile helpers from the provider context. They may
  report auth failures and request profile rotation or fallback, but they do
  not read global env or settings directly.
- Agent runtime adapters own native progress/reasoning normalization. The turn runner sees
  normalized reasoning, assistant text, tool events, usage, and terminal
  outcome records.
- Runtime state is generic `conversation_provider_state` metadata, not
  hardcoded conversation columns.
- Runtime lifecycle is reload-aware: new sessions use the new runtime
  snapshot, while in-flight sessions finish against their captured snapshot.

### Agent Profiles And Delegated Jobs

Agent profiles are reusable agent definitions. They are the Pawrrtal equivalent
of Claude Code-style subagent files: named instructions plus metadata that tells
the Paw when to delegate, which tools are allowed, which runtime/model is
preferred, and whether memory is available. A plugin may contribute an
`agent_profile`, but the plugin does not implement the subagent runtime.

```json
{
  "type": "agent_profile",
  "id": "code_reviewer",
  "title": "Code Reviewer",
  "description": "Review code changes for correctness, maintainability, and missing tests.",
  "instructions": "agents/code_reviewer.md",
  "tools": ["read_file", "grep", "capabilities_search"],
  "runtime_preferences": ["openai_codex_runtime", "claude_agent_runtime"],
  "model_preferences": ["openai-codex:openai/gpt-5.5-codex"],
  "slots": ["code_review", "architecture_review"],
  "memory": {
    "scope": "workspace",
    "enabled_by_default": false
  }
}
```

Rules:

- Agent profiles are capability-like metadata, not adapters. They are searchable,
  selectable, slotted, and inspectable, but they do not execute code.
- Delegated agent jobs are created by the kernel through an explicit job API.
  The job captures one registry snapshot, one profile snapshot, and one runtime
  adapter snapshot.
- Plugins may request delegated jobs through kernel tools or APIs, but they do
  not spawn arbitrary background processes or own job persistence.
- Profile hot reload is allowed. New jobs use the new profile snapshot; active
  jobs continue with the captured profile and runtime.
- Profile tool lists are allowlists within the current workspace policy. A
  plugin-contributed profile cannot grant a tool, MCP server, filesystem access,
  or permission mode the user/workspace has not allowed.
- Automatic delegation uses profile `description`, `slots`, `tags`, and runtime
  health. Explicit invocation by profile id always wins over automatic matching.
- Plugin-contributed profiles ignore unsafe runtime-control fields unless the
  profile is copied into a trusted repo/user scope.
- The UI may call these "subagents" for familiarity, but internal contracts use
  `agent_profile` and `delegated_agent_job`.

## Registry Snapshot Interface

`PluginHost` is the only long-lived mutable object. It owns the current
immutable `PluginRegistrySnapshot`.

Required public interface:

```python
class PluginHost:
    def current(self) -> PluginRegistrySnapshot: ...
    async def reload(self, request: ReloadRequest) -> ReloadResult: ...
    async def validate(self, request: ValidateRequest) -> ValidateResult: ...
    async def enable(self, request: EnablePluginRequest) -> PluginStateView: ...
    async def disable(self, request: DisablePluginRequest) -> PluginStateView: ...
```

Snapshot interface:

```python
class PluginRegistrySnapshot:
    fingerprint: str
    created_at: datetime
    plugins: Mapping[str, PluginRuntime]
    load_outcomes: Mapping[str, PluginLoadOutcome]
    tools: tuple[ToolContribution, ...]
    turn_context_providers: tuple[TurnContextProviderContribution, ...]
    turn_observers: tuple[TurnObserverContribution, ...]
    agent_runtimes: Mapping[str, AgentRuntimeContribution]
    agent_profiles: Mapping[str, AgentProfileContribution]
    providers: Mapping[str, ProviderContribution]
    channels: Mapping[str, ChannelContribution]
    migration_providers: Mapping[str, MigrationProviderContribution]
    routers: tuple[RouterContribution, ...]
    settings: tuple[SettingsContribution, ...]
    skills: tuple[SkillContribution, ...]
```

Snapshot rules:

- Snapshots are immutable.
- Callers capture a snapshot once per operation.
- Reload builds a candidate snapshot without mutating the current snapshot.
- Reload publishes the candidate atomically only after validation and required
  lifecycle startup succeeds.
- Old snapshots remain valid until in-flight operations finish.
- Load outcomes are part of the snapshot so API, CLI, frontend, and Paw tools
  explain the same state without re-running discovery.
- Manifest and discovery snapshots are explicit values passed through a flow.
  Do not add hidden global manifest caches or wall-clock TTLs. Runtime module
  caches are allowed only after plugin code or installed artifacts are loaded.

## Hot Reload Algorithm

Reload implementation in `backend/app/plugins/reload.py`:

1. Acquire the host reload lock.
2. Discover manifests from bundled, global, and workspace roots.
3. Parse manifests and collect structural errors.
4. Resolve overrides and dependencies.
5. Read state files and compute active/inactive/misconfigured/failed states.
6. Run validation commands for changed or explicitly requested plugins.
7. Build a candidate registry snapshot.
8. Start lifecycle instances for new or changed provider/channel/scheduler
   capabilities that require startup.
9. If startup fails, stop newly started lifecycle instances and reject reload.
10. Publish the new snapshot.
11. Stop lifecycle instances that disappeared or were replaced.
12. Emit audit events for reload start, reload success/failure, and every
    plugin state transition.

Development watcher:

- Implement `paw plugins watch` by calling `watchfiles` if available.
- Watch only plugin roots and state directories.
- Debounce reloads by 250 ms.
- Print progress to stderr.
- Emit reload results to stdout only in `--json` mode.

Production:

- No background watcher by default.
- Explicit API/CLI reload is supported.

## Channel Reload Matrix

Channel reload behavior is intentionally narrower than generic manifest reload.

| Change | Behavior |
|---|---|
| Enable/disable channel plugin | Hot reloads the registry snapshot. New ingress requests and new deliveries use the new state. Disabled channels return `404` or `409` with a clear diagnostic. |
| Workspace-scoped channel setting/env value | Hot reloads config state. If the value affects a running client, reload starts a new channel lifecycle instance before publishing the snapshot, then stops the old instance. |
| Message action metadata, schema fragments, labels, or media-source params | Hot reloads for future message-tool discovery calls. In-flight turns keep their captured action set. |
| Delivery formatting implemented as manifest/CLI/config metadata | Hot reloads for future deliveries. |
| Python channel code in development | Dev-only re-import may apply on reload when the file watcher is enabled. Failed import keeps the old snapshot active. |
| Python channel code in production | Requires process restart. Reload reports `restart_required` instead of pretending code changed. |
| Stable dispatch handler behavior behind `/{surface}/ingress/{path}` | Hot reloads only when the behavior is manifest/CLI/config based, or dev-only Python re-import is enabled. |
| FastAPI route shape, new HTTP method, new top-level prefix, or removed route | Requires process restart. Channel plugins should use the stable dispatch route when they need reloadable ingress behavior. |
| Polling/webhook client lifecycle config | Reload starts new instance, publishes snapshot after startup succeeds, then stops old instance. Failed startup leaves the old instance active. |
| External webhook registration at the provider | Not implicit. Doctor/setup must report the required external change and the user/operator applies it explicitly. |

## Rollout Rules

- Keep PRs reviewable. Prefer one migration slice per PR after the foundation
  lands.
- Commit after each coherent interface/test milestone.
- Every PR touching the plugin platform includes:
  - the slice number;
  - the Paw CLI commands run;
  - the path to the `paw plugins proof collect` artifact;
  - the backend/frontend gates run;
  - whether hot reload was proved or not relevant.
- CI runs `paw plugins proof check --changed-from origin/development` for
  plugin-platform paths. A reviewer may waive it only when the PR changes docs
  that explicitly do not alter implementation behavior.
- Do not merge a slice that leaves a mixed static/dynamic caller path unless
  the next slice removes it and the current PR has tests around the transition.
- Do not describe failures as unrelated or pre-existing. Fix warnings and
  broken tests in touched surfaces, or split a sibling PR when the fix is truly
  separate.

## Completion Criteria

The platform is complete when all of these are true:

- A workspace-authored CLI plugin can be created, validated, enabled, reloaded,
  and called by the Paw without backend restart.
- Notion is a manifest-backed CLI plugin.
- Active recall is a manifest-backed turn context provider.
- Providers are plugin adapter contributions, and model catalog rows come from provider
  manifests.
- Session-owning runtimes such as OpenAI Codex are exposed through
  `agent_runtime` adapters, not provider-specific core protocol branches.
- Agent profiles are plugin-contributed definitions, and delegated agent jobs
  are kernel-owned executions through `agent_runtime` adapters.
- Channels are plugin adapter contributions, and Telegram/Google Chat lifecycle is
  controlled by plugin enablement and doctor status.
- Channels expose channel-specific actions through the shared message tool.
- Channel and provider conversation state is generic.
- Migration/import flows are plugin capabilities with detect, plan, apply, and
  report behavior.
- The plugin settings UI is driven by plugin metadata.
- The Paw CLI is the standard implementation and verification surface for
  plugin work.
- The old static plugin registry, static provider factory tables, static
  channel registry, and static env metadata mirrors are gone.
