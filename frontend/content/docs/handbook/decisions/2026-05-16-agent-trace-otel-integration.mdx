# ADR: Emit OpenTelemetry GenAI Spans For Agent Tracing

- **Date:** 2026-05-16
- **Owners:** OctavianTocan
- **Tracking:** Agent trace integration.

<Callout type="info" title="Status: Accepted">
Add `app/infrastructure/observability/agent_trace.py`, wire it into
`turns/pipeline.run_turn` and model tool dispatch, and emit
standard OTel spans. Local dev can point `OTEL_EXPORTER_OTLP_ENDPOINT`
at a collector on `http://localhost:5899`; the same spans route to
Grafana, Honeycomb, SigNoz, or another OTel backend in deployed
environments.
</Callout>

## Context

Local OTLP collectors can render an agent's tokens, tool calls, and turn
boundaries live. The wire format should stay generic: OTLP/HTTP-JSON
with **OpenTelemetry GenAI / Traceloop** attributes from the `gen_ai.*`
semantic-convention family.

Pawrrtal already has the OTel SDK + OTLP/HTTP exporter wired in
`backend/app/infrastructure/telemetry/` (gated by `OTEL_EXPORTER_OTLP_ENDPOINT`).
The chat pipeline has three architectural seams that map cleanly to the
span types:

1. **Turn root** — `turns/pipeline.run_turn` is the single
   per-turn entry point shared by all surfaces (HTTP `/api/chat`,
   Telegram, voice).
2. **LLM call** — same site; `turn_orchestrator` already provides an
   `EventHook = Callable[[StreamEvent], list[StreamEvent]]` seam so a
   read-only sidecar consumer can observe the provider stream without
   changing the channel deliverer.
3. **Tool execution** — `run_model_tool_loop._execute_and_log_tool_call` is the
   single dispatch site for every tool the agent calls (enforced by
   `scripts/check-no-tools-in-providers.py`).

## Decision

Emit OpenTelemetry GenAI / Traceloop-shaped spans, not a vendor-specific wire
protocol. Local collectors and deployed OTel backends consume the same
spans without change.

### Schema we emit

The schema follows OpenTelemetry GenAI attributes, with Traceloop-compatible
fields where collectors still expect them.

- **`pawrrtal.turn`** — root span per chat turn.
  Attributes: `pawrrtal.conversation_id`, `pawrrtal.user_id`,
  `pawrrtal.surface`, `pawrrtal.request_id`, `gen_ai.request.model`.
- **`pawrrtal.llm.chat`** — one per LLM call.
  Discriminator `traceloop.span.kind = "llm"`. Attributes include
  `gen_ai.operation.name = "chat"`, `gen_ai.request.model` /
  `gen_ai.response.model`, `gen_ai.input.messages` (JSON),
  `gen_ai.output.messages` (JSON, stamped on flush),
  `gen_ai.system_instructions` (JSON, optional),
  `gen_ai.usage.input_tokens` / `.output_tokens`,
  `gen_ai.response.finish_reasons`. Span events
  `gen_ai.content.delta` / `gen_ai.thinking.delta` per streamed chunk.
- **`<tool_name>.tool`** — one per tool execution.
  Discriminator `traceloop.span.kind = "tool"`. Attributes
  `traceloop.entity.name`, `traceloop.entity.input` (JSON args),
  `traceloop.entity.output` (JSON result), `otel.status.message` on
  failure.

### Wiring

- `backend/app/infrastructure/observability/_schema.py` — attribute-name
  constants + payload renderers (`render_input_messages`).
- `backend/app/infrastructure/observability/_recorders.py` — `LLMSpanRecorder` +
  `ToolSpanRecorder` (mutable per-span state).
- `backend/app/infrastructure/observability/agent_trace.py` — public context
  managers (`turn_span`, `llm_span`, `tool_span`) + `agent_event_hook`
  that mirrors `StreamEvent`s onto a recorder via a flat dispatch table.
- `backend/app/turns/pipeline/runner.py` — `run_turn` opens
  `turn_span` + `llm_span`, registers `agent_event_hook` ahead of
  the caller's hooks, stamps usage / stop-reason on the LLM span at
  finalisation.
- `backend/app/agents/model_tool_loop/tool_calls.py` — wraps the dispatch
  in `tool_span`.

### Configuration

```bash
# Local dev with a collector
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:5899
OTEL_SERVICE_NAME=pawrrtal-backend
```

When the endpoint env var is unset every span is a no-op
(`trace.get_tracer` returns a no-op tracer), so production installs
without OTel pay nothing.

## Why this shape

1. **Open standard, not a vendor protocol.** The Traceloop attribute
   schema is the OTel GenAI semantic convention. Grafana, Honeycomb,
   SigNoz, Phoenix, and local collectors all parse the same shape. No
   second instrumentation pass when we add a remote OTel backend.
2. **Three sites, three layers.** The instrumentation matches the
   chat pipeline's existing seams (router → channel → loop). No
   touching the providers — that surface stays restricted, and the
   `no-tools-in-providers` rule keeps the tool span centralised in
   `run_model_tool_loop`.
3. **`EventHook` is the right seam.** `turn_orchestrator` already designed
   for non-destructive sidecar emission via
   `EventHook = Callable[[StreamEvent], list[StreamEvent]]`. The
   agent trace hook returns `[]` so the chat aggregator + channel
   deliverer never see the observability layer.

## Alternatives considered

- **Vendor-specific live-event SDKs.** They require a second protocol on
  top of OTel and tie traces to one viewer. Rejected: locks us to one
  debugging surface.
- **Custom non-Traceloop attributes.** Would require a parallel adapter.
  Rejected for the same reason.
- **Patching `providers/` directly.** Forbidden by
  `.claude/rules/architecture/no-tools-in-providers.md` and the
  CODEOWNERS scope on the provider files.

## Tests

`backend/tests/test_observability_agent_trace.py` runs against an
in-memory OTel `TracerProvider` + `InMemorySpanExporter` and asserts
the exact agent trace attribute shape (`traceloop.span.kind`,
`gen_ai.input.messages`, etc.) for every code path. 11 tests; ~120ms
total. Any future schema change surfaces as a test failure here rather
than silently breaking trace rendering.

## Follow-ups

- `/setup-agent-replay` scaffold: ingest a captured trace
  and re-run it through `run_turn` via `ScriptedStreamFn` for live

- Promote the `claude_provider.py` CLI stderr buffer into the LLM

  token output in the trace timeline.
