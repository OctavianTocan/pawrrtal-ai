# skill-gen

Code-generation tool that scans source files for embedded skill fragments and assembles them into standard `SKILL.md` files in a configured output folder.

## Motivation

Agent skill definitions can grow large and disconnect from the code they describe. By co-locating skill fragments next to the source they relate to using `//<skill-gen>` markers, teams keep agent knowledge close to the code while a single generator merges everything into valid `<output>/<name>/SKILL.md` files.

## How it works

### 1. Scan

Recursively walk a **base folder** (default: repo root) and find every file that contains the marker comment:

```
//<skill-gen>
```

Everything between a `<skill-gen>` opening tag and its corresponding `</skill-gen>` closing tag is treated as a **skill fragment**. A single file may contain multiple fragments.

```ts
// some-module.ts

//<skill-gen>
// ---
// name: effect
// description: "Effect-TS patterns for type-safe applications."
// ---
//
// ## Service Pattern
//
// Always use `Effect.Service` for application services:
// ```typescript
// class MyService extends Effect.Service<MyService>()("MyService", { ... }) {}
// ```
//</skill-gen>
```

Comment prefixes (`//`, `#`, etc.) at the start of each line inside the block are stripped before parsing.

### 2. Parse

Each extracted block is parsed as **Markdown with YAML frontmatter**. The parser must:

- Strip leading comment characters (`//` or `#`) and an optional single space from every line.
- **Expand template variables** (see **Template variables** below).
- Extract YAML frontmatter (between `---` delimiters) and parse it.
- Require a `name` key in the frontmatter to identify which skill this fragment belongs to.
- Require a `description` key in the frontmatter (used for the skill's description field).
- Preserve any extra frontmatter lines (for example `paths:`) in the generated output.
- The remainder after frontmatter is the **markdown body**.

### Template variables

Before parsing, the following template variables are expanded in each fragment:

| Variable | Expands to | Example |
|----------|------------|--------|
| `$$file` | Relative path from repo root to the source file | `packages/server/api/src/server.ts` |
| `$$directory` | Relative path from repo root to the source file's directory | `packages/server/api/src` |

### 3. Accumulate / Merge

Fragments that share the same `name` are **merged** into a single skill document. Merge rules:

| Field | Strategy |
|-------|----------|
| `name` | Must match (this is the merge key) |
| `description` | Last writer wins (with warning if different) |
| extra frontmatter | Preserved until a later fragment declares extra frontmatter; conflicting non-empty values warn and later wins |
| markdown body | **Concatenated** in source-file order (sorted by relative path for determinism) |

Each merged skill also tracks which source files contributed to it (for the generated header comment).

### 4. Output

Write each merged skill to the **output folder** (default: `.agents/skills/`; Pawrrtal's root scripts pass `.cursor/plugins/pawrrtal/skills`) as `<name>/SKILL.md`.

The output format is:

```markdown
---
name: my-skill
description: "Description of the skill."
paths:
  - "packages/server/**/*.ts"
---

<!-- AUTO-GENERATED by skill-gen -- DO NOT EDIT -->
<!-- Source: packages/server/api/src/server.ts -->
<!-- Source: packages/server/api/src/handlers.ts -->

# Heading

Body content from all merged fragments...
```

The frontmatter must be the first thing in the file because Claude Code's skill loader requires `SKILL.md` files to start with `---`.

If a generated file would overwrite a file that was **not** previously generated by this tool (i.e. missing the `AUTO-GENERATED` header), abort with an error. This prevents accidentally overwriting hand-written skill files.

## CLI

Entrypoint: `index.ts` (bun/ts).

```
Usage: skill-gen <command> [options]

Commands:
  generate           Scan, merge, and write skill files (default command)
  check              Dry-run: exit 1 if generated output differs from existing files
  e2e-test           Run e2e tests against fixtures

Options:
  --base <path>      Base folder to scan (default: repo root, detected via git)
  --output <path>    Output folder for generated skills (default: <base>/.agents/skills)
  --verbose          Print discovered fragments and merge diagnostics
  --help             Show help
```

### `check` command

Useful in CI to verify that committed skill files are up-to-date with their source fragments. Exits 0 if everything matches, 1 if there are differences (with a diff printed to stderr).

From the repo root:

```bash
bun run skill-gen:check
```

### `generate` command

Runs the full pipeline: scan -> parse -> merge -> output. Writes generated skill files to the output directory.

From the repo root:

```bash
bun run skill-gen:generate
```

  e2e-test/             test input fixtures (source files with markers)
  e2e-test-expected/    expected generated skill output
```

## E2E testing

`index.ts` exposes an **e2e-test** command (e.g. `bun run e2e-test`) that:

1. Runs the generator against `e2e-test/` as the base folder.
2. Compares output to `e2e-test-expected/`.
3. Exits 0 on match, 1 on diff (printing the diff).

### E2E fixtures

The e2e test suite should include:

1. **Single-source skill**: one file contributes one skill fragment -> one SKILL.md
2. **Multi-source merge**: two files contribute fragments with the same skill name -> merged SKILL.md with concatenated body
3. **Multiple skills**: one file with two different skill fragments -> two separate SKILL.md files

## Edge cases & decisions

- **Nested markers**: not supported. If `<skill-gen>` appears inside an already-open block, emit an error.
- **Empty blocks**: skip silently.
- **Encoding**: UTF-8 only.
- **Binary files**: skip files that fail UTF-8 decode.
- **Symlinks**: follow symlinks (default `fs` behavior).
- **Ignore patterns**: skip `node_modules/`, `.git/`, `vendor/`, `.context/`, and the output folder itself.
- **Template variables in non-repo context**: if the tool cannot detect a git repo root (e.g. when scanning a standalone directory), `$$file` and `$$directory` resolve relative to the `--base` folder instead.
- **Hand-written skills**: existing `.agents/skills/<name>/SKILL.md` files that do NOT have the `AUTO-GENERATED` header are left untouched. The generator only manages files it created.
- **References subdirectory**: the generator does NOT manage `references/` subdirectories. Those remain hand-written. Only `SKILL.md` is generated.
