# Quickstart: Validate The Effect Paw CLI Plan

This guide describes the expected validation flow after the feature is implemented.

## Prerequisites

- Run from the repository root.
- Dependencies installed with the repo's normal install flow.
- `backend/vendor/effect-smol` is present for Effect v4 source checks.

## 1. Verify The New Package Exists

```bash
test -f packages/paw-cli/package.json
test -f packages/paw-cli/src/Main.ts
test -f packages/paw-cli/src/Cli.ts
test -f packages/paw-cli/src/Commands.ts
bun run --filter '@pawrrtal/cli' typecheck
```

Expected outcome:

- The package exists at `packages/paw-cli`.
- The package uses the comcom-style CLI layout: executable entrypoint, root wiring, command registry, helpers, infrastructure, and `Modules/<Name>/Command.ts` command groups.
- TypeScript 6.x typecheck passes without starting frontend or backend dev servers.

Optional TypeScript-Go comparison, once the dependency is available:

```bash
bun run --filter '@pawrrtal/cli' typecheck:tsgo
```

Expected outcome:

- This optional check reports the same package diagnostics as the canonical TypeScript check before it is considered for CI.

## 2. Verify Root CLI Behavior

```bash
bun run --filter '@pawrrtal/cli' start -- --help
bun run --filter '@pawrrtal/cli' start -- --version
bun run --filter '@pawrrtal/cli' start -- doctor --json
bun run --filter '@pawrrtal/cli' start -- context --json
bun run --filter '@pawrrtal/cli' start -- whoami --json
bun run --filter '@pawrrtal/cli' start -- completions bash
bun run --filter '@pawrrtal/cli' start -- completions zsh
```

Expected outcome:

- Help lists only intentionally implemented new CLI commands.
- Version prints successfully.
- `doctor --json` includes pass/warn/fail counts.
- `context --json` prints active context without secrets.
- `whoami --json` returns the same active identity/context meaning as `context --json`.
- Completion generation writes shell completion text.

## 3. Verify Output Modes

```bash
bun run --filter '@pawrrtal/cli' start -- context
bun run --filter '@pawrrtal/cli' start -- --profile local context
bun run --filter '@pawrrtal/cli' start -- context --json
bun run --filter '@pawrrtal/cli' start -- context --plain
```

Expected outcome:

- Human output is readable.
- `--profile local` changes only that invocation's resolved profile.
- JSON output is parseable.
- Plain output is tab-separated without headers.
- Progress, warnings, and errors do not pollute parseable stdout.

## 4. Verify Input Source Rejection

Use the first command added by implementation that accepts document/body input.

```bash
bun run --filter '@pawrrtal/cli' start -- <command> --content 'body' --file ./body.md
```

Expected outcome:

- Command exits with code `2`.
- Error explains that only one body source is allowed.
- Error names the accepted alternatives.

## 5. Verify Generated Skills

```bash
bun run skill-gen:generate
bun run skill-gen:check
test -f .agent/skills/paw/SKILL.md
test -f .agent/skills/domain-cli/SKILL.md
rg -n 'AUTO-GENERATED by skill-gen' .agent/skills/paw/SKILL.md .agent/skills/domain-cli/SKILL.md
```

Expected outcome:

- `skill-gen:check` passes.
- Both generated skills exist under `.agent/skills`.
- Both skills have the generated header.

## 6. Verify Old Python CLI Removal

```bash
test ! -d backend/app/cli/paw
test ! -d backend/tests/paw
test ! -d backend/tests/e2e_paw
! rg -n 'app\.cli\.paw|uv run paw|backend/app/cli/paw|backend/tests/paw|paw-extend' scripts justfile backend/pyproject.toml .agent/skills package.json
```

Expected outcome:

- Old Python Paw CLI directories are gone.
- Supported launchers no longer execute `uv run paw`.
- Generated skills do not cite deleted Python CLI files.
- `paw-extend` is gone or no longer generated from old Python CLI fragments.

## 7. Verify Focused Gates

```bash
bun run --filter '@pawrrtal/cli' check
bun run skill-gen:check
UV_CACHE_DIR=/tmp/pawrrtal-uv-cache just check
```

Expected outcome:

- CLI package gate passes without frontend build/browser work.
- Skill drift check passes.
- Repo check passes.

## 8. Optional Full Gate

Use only when the implementation touches broader repo behavior.

```bash
just typecheck
just test
```

Expected outcome:

- Full type and test gates pass when their scope is intentionally selected.
